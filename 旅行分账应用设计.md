# 旅行分账应用 — 设计文档

## 1. 产品定位

一句话：**一个旅行团体的「共享账本 + 行程回忆册」**。

使用主线：建行程 → 拉成员 → 沿途记支出、拍照片 → 行程结束一键结算。

竞品参照：Splitwise（分账）、Tricount（旅行分账）、相册类 App（回忆）。差异点：把「算账」和「回忆」合在同一条行程时间线里——每笔支出天然挂在某个地点、某一天上。

用户真正的痛点是**旅行结束后算账麻烦、容易伤感情**，所以分账是核心，地点和照片是让账目有归属感的附属功能。

---

## 2. 核心概念（数据模型）

### Trip 行程
| 字段 | 说明 |
|------|------|
| 名称、日期范围 | 「2026 日本五日游」 |
| 主货币 | 结算货币，如 MYR。所有余额和结算建议都折算到主货币 |
| 成员列表 | 见 Member |
| 邀请码 | 别人凭码/链接加入，不需要好友系统 |

### Member 成员
- 昵称、头像。
- **关键设计：允许「虚拟成员」**——没装 App、没注册的人也能被记账（记账人替他记），之后他注册了可以「认领」这个身份。这避免了"必须全员先注册才能用"的冷启动死穴，是这类产品能不能被用起来的关键。

### Stop 地点/站点
- 名称、坐标（可选）、到达时间、照片若干、文字游记。
- 按时间排列构成行程时间线。支出可以挂在某个 Stop 下，也可以不挂（如"给全程买的保险"）。

### Expense 支出（最重要的实体）
| 字段 | 说明 |
|------|------|
| 金额 | **以最小货币单位的整数存储**（分、日元），绝不用浮点数 |
| 原始货币 + 汇率 | 记账那一刻锁定汇率并存下来，可手动修改。不用结算时的实时汇率——避免"上周记的账今天汇率变了"的争议 |
| 付款人 | 谁垫的钱；支持多人合付（如房费两人各付一半） |
| 参与人 + 分法 | **每笔独立选择参与人，不强制全员平分**（"这顿饭 B 没吃"是高频场景）。分法：平分（默认）/ 按金额 / 按比例 / 按份数 |
| 类别 | 餐饮 / 住宿 / 交通 / 门票 / 购物 / 其他 |
| 所属 Stop | 可选 |
| 备注、小票照片 | 可选 |
| 记录人、修改历史 | 每笔账可追溯谁记的、改过什么——分账纠纷的最终裁判 |

### Settlement 结算记录
- A → B，金额（主货币），是否已还。结算建议生成后，实际转了账就标记，余额相应清零。

### 不变量（系统自检）
- **所有成员净余额之和恒等于 0。** 任何时刻这条不成立就是有 bug。这是整个账本的对账基础。

## 3. 结算算法

**第一步：算净余额。** 每人净余额 = 垫付总额 − 应摊总额（全部按各笔锁定的汇率折算到主货币）。正数 = 别人欠他，负数 = 他欠别人。

**第二步：最少转账。** 贪心法：每轮取最大债权人和最大债务人配对，冲销较小的一方，循环直到全部清零。n 个人最多 n−1 笔转账结清。

> 注：理论上的"绝对最少笔数"是 NP-hard 问题，贪心不保证最优，但实践上（旅行团 ≤10 人）结果足够好，且规则简单可解释。

**除不尽的处理。** 100 元 3 人平分 = 33.33…，余数（1 分）**由付款人吸收**。规则写死、在界面上展示出来，不给用户配置项——分账产品最怕的是"说不清"，不是"少收一分钱"。

## 4. 同步架构（多人共享）

**推荐用现成后端（Supabase，其次 Firebase），不自建服务器。** 理由：账号系统、数据库、实时订阅、文件存储（照片）全都自带，把复杂度压到只写客户端逻辑。自建后端对这个产品阶段是纯负担。

**离线优先。** 旅行场景经常没网（尤其出国、山区）：
- 本地先写、立即可见，有网后台同步。
- 冲突天然少：账本是「各人添加自己的记录」模型，两个人同时改同一条的概率很低。
- 真冲突时：last-write-wins + 保留修改历史（谁在什么时候改成了什么），出争议翻历史。

**加入机制。** 行程邀请码 / 分享链接，点开即入。不做好友系统、不做通讯录导入——都是 V1 用不上的重量。

## 5. 功能分期

| 版本 | 内容 | 为什么先做它 |
|------|------|------------|
| **V1 单机可用** | 建行程、加成员（含虚拟成员）、记支出（多币种、自选参与人）、结算建议，数据存本地 | **把账算对是信任基础。** 一个人记账、结算时给大家看，已经解决 80% 的痛点 |
| **V2 多人同步** | 账号、邀请码、实时同步、离线队列 | 同步是传播基础——每个成员都装了，产品才会自己长 |
| **V3 回忆与统计** | 地点时间线、照片游记、按类别/按人/按天的统计图表、行程结束总结页 | 回忆是留存基础——行程结束后还有理由打开它 |

顺序不能反：账算错一次，用户就再也不信了；没人用的时候做同步是白做；没数据的时候做图表是空的。

## 6. 主要界面（文字线框）

```
行程列表页
 └─ 行程主页（底部 4 个 Tab）
     ├─ 账本：按天分组的支出流水，右下角大按钮「记一笔」
     ├─ 时间线：地点 + 照片 + 游记，按时间排
     ├─ 结算：每人净余额条形图 + 「谁转谁多少」清单 + 标记已还
     └─ 统计：按类别饼图、按天柱状图、按人汇总
```

**「记一笔」是核心高频操作，必须 ≤10 秒完成：**
金额（数字键盘直接弹出）→ 货币（默认上次用的）→ 付款人（默认自己）→ 参与人（默认全选）→ 类别（一排图标点一下）→ 保存。备注、照片、挂地点都是可选的折叠项。

## 7. 已知风险与取舍

| 风险 | 对策 |
|------|------|
| 多人同步复杂度高一个量级 | 用 BaaS 压成本；V1 先不做 |
| 汇率争议 | 记账时锁定汇率、可手改，结算不用实时汇率 |
| 分账纠纷 | 每笔可追溯（记录人 + 修改历史）；余数规则写死并展示 |
| 照片存储成本 | V3 再考虑：客户端压缩、每 Stop 限张数 |
| 全员必须注册才能用 | 虚拟成员 + 事后认领 |

## 8. 验证算例：3 人日本 5 日游

成员：A、B、C。主货币 MYR。

| 支出 | 原始金额 | 汇率（锁定） | 折合 MYR | 付款人 | 参与人 |
|------|---------|------------|---------|--------|--------|
| 晚餐 | ¥30,000 | 1 JPY = 0.030 MYR | 900.00 | A | A、C（B 没吃） |
| 民宿 | RM 1,200 | — | 1,200.00 | C | A、B、C |
| 车票 | ¥9,000 | 1 JPY = 0.031 MYR | 279.00 | B | A、B、C |

**应摊**（单位：仙，MYR 分）：
- 晚餐 90,000 仙 ÷ 2 = 各 45,000，整除无余数 → A 摊 450.00，C 摊 450.00
- 民宿 120,000 仙 ÷ 3 = 各 40,000，整除 → 每人 400.00
- 车票 27,900 仙 ÷ 3 = 各 9,300，整除 → 每人 93.00

**净余额** = 垫付 − 应摊：
| 成员 | 垫付 | 应摊 | 净余额 |
|------|------|------|--------|
| A | 900.00 | 450 + 400 + 93 = 943.00 | **−43.00** |
| B | 279.00 | 0 + 400 + 93 = 493.00 | **−214.00** |
| C | 1,200.00 | 450 + 400 + 93 = 943.00 | **+257.00** |

自检：−43 − 214 + 257 = 0 ✓

**结算建议（贪心）**：最大债务人 B → 最大债权人 C 转 **RM 214.00**；剩 A → C 转 **RM 43.00**。两笔转账结清（3 人，≤ n−1 = 2 笔 ✓）。

该算例覆盖了：多币种 + 不同汇率、非全员参与、跨货币付款人，模型均无歧义处理。
